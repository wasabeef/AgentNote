## Quick Setup

Add this workflow — created automatically by `agent-note init`:

```yaml
# .github/workflows/agentnote.yml
name: Agent Note
on:
  pull_request:
    types: [opened, synchronize]
permissions:
  contents: read
  pull-requests: write
jobs:
  report:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - uses: wasabeef/AgentNote@v0
```

That's it. Every PR gets an AI session report.

## Inputs

| Input | Default | Description |
|---|---|---|
| `base` | PR base branch | Base branch to compare against |
| `pr_output` | `description` | PR report destination: `description`, `comment`, or `none` |
| `dashboard` | `false` | Write/update dashboard snapshot files for the current PR |
| `dashboard_dir` | `packages/dashboard/public` | Parent directory where `notes/*.json` will be written |
| `dashboard_url` | — | Public dashboard URL to link from the PR report after the dashboard is already deployed |
| `output` | — | Legacy alias for `pr_output` (`description` or `comment`) |
| `comment` | `true` | Legacy: set to `false` to disable PR report posting |

### Post as comment instead of description

```yaml
- uses: wasabeef/AgentNote@v0
  with:
    pr_output: comment
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

### Generate dashboard data alongside the PR report

```yaml
- uses: wasabeef/AgentNote@v0
  id: agent-note
  with:
    pr_output: description
    dashboard: true
    dashboard_dir: packages/dashboard/public
    # Optional. Set this only after the dashboard is publicly available.
    dashboard_url: https://docs.example.com/dashboard/
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

Use `packages/dashboard/public` as the parent directory; the action creates or updates `packages/dashboard/public/notes/*.json`. When dashboard notes were written, the PR report can also add an `Open dashboard` link near the top if you explicitly provide `dashboard_url`.

## Outputs

| Output | Type | Description |
|---|---|---|
| `overall_ai_ratio` | number | PR-wide AI ratio (0-100) |
| `overall_method` | string | `line`, `file`, `mixed`, or `none` |
| `tracked_commits` | number | Commits with agent-note data |
| `total_commits` | number | Total commits in PR |
| `total_prompts` | number | Total prompts across all commits |
| `json` | string | Full structured report |
| `markdown` | string | Rendered markdown report |
| `dashboard_dir` | string | Directory where the dashboard bundle was written |
| `dashboard_commits` | number | Number of tracked commits written to the dashboard bundle |
| `dashboard_url` | string | Public dashboard URL linked from the PR report when explicitly configured |

### Use outputs in workflow

```yaml
- uses: wasabeef/AgentNote@v0
  id: agent-note
  with:
    base: main

- run: echo "AI ratio: ${{ steps.agent-note.outputs.overall_ai_ratio }}%"
```

## Report Format

The PR report includes:

- **Header** — `Total AI Ratio` with a progress bar, plus a separate plain-text `Model` line
- **Dashboard link** — added only when the action writes dashboard notes and you explicitly configure `dashboard_url`
- **Table** — per-commit AI ratio, prompt count, and files with attribution
- **Prompts** — collapsible section with per-commit prompt/response pairs

<Aside type="tip">
  Commit hashes in the report are linked to their GitHub commit page.
</Aside>

## Notes Sharing

The action fetches `refs/notes/agentnote` from the remote. The `pre-push` git hook installed by `agent-note init` auto-pushes notes on every `git push` — no manual push needed.

## Dashboard Package

The interactive dashboard now lives in `packages/dashboard/` and is deployed alongside the docs site under `/dashboard/`. The action can update the bundle files in that package, while your GitHub Pages workflow decides when to publish them.

The repository does not ship sample dashboard JSON. A fresh dashboard stays empty until your Pages sync workflow restores or generates `packages/dashboard/public/notes/*.json`.

`packages/dashboard` keeps its local default at `/`, so your Pages workflow should inject deploy-specific values with `SITE`, `BASE`, `PUBLIC_REPO`, and optionally `PUBLIC_REPO_URL`.

## GitHub Pages Sync

For the live dashboard, split responsibilities by event:

- `gh-pages/dashboard/notes/*.json` is the durable data store
- each run restores those files into `packages/dashboard/public/notes/`
- `pull_request` (`opened`, `reopened`, `synchronize`) rewrites the current PR's note set and persists it back to `gh-pages`
- `push` on `main` finalizes matching PR notes as `merged`, rebuilds the site, and deploys the shared Pages URL
- dashboard builds should pass repo-specific env vars such as `SITE=https://<owner>.github.io`, `BASE=/<repo>/dashboard`, and `PUBLIC_REPO=<owner>/<repo>`
- a brand-new repo can accumulate dashboard note data before the first production deploy, but the public Pages URL only appears after the first `main` deployment

Recommended flow:

1. Restore `gh-pages/dashboard/notes` into `packages/dashboard/public/notes`.
2. On `pull_request` (`opened`, `reopened`, `synchronize`), remove the old note files for that PR number, then rewrite the note JSON for the PR's current commits.
3. Persist the updated `dashboard/notes` back to `gh-pages`.
4. On `push` to `main`, write note JSON for the pushed commits, update matching PR note state to `merged`, then build and deploy the Pages artifact.

This keeps `main` free of generated JSON while still letting one shared dashboard show open and merged PR history.
