## Cosa ottieni

Il Dashboard offre al tuo team un'unica URL condivisa per la cronologia di Agent Note.

- Vedere la cronologia delle PR aperte e mergeate in un solo posto
- Controllare commit, prompt e attribution senza aprire ogni PR
- Condividere una sola URL GitHub Pages, di solito sotto `/dashboard/`

Usalo quando il PR report non basta più e vuoi una vista condivisa per il team.

## Prima di iniziare

Ti serve il Dashboard solo se vuoi una vista web condivisa.

- Se ti bastano gli AI session report dentro le pull request, fermati a [GitHub Action](./github-action/).
- Se vuoi una vista Pages condivisa per la cronologia open e merged, continua qui.

## Quick Setup

Per prima cosa, fai in modo che la Action scriva i Dashboard bundle per ogni PR:

```yaml
- uses: wasabeef/AgentNote@v0
  id: agent-note
  with:
    pr_output: description
    dashboard: true
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

Questo crea o aggiorna `packages/dashboard/public/notes/*.json` per la PR corrente. Poi il tuo Pages workflow può ripristinare, persistere e pubblicare quei file.

## URL pubblica

Di solito il Dashboard viene pubblicato qui:

- `https://<owner>.github.io/<repo>/dashboard/`

Per esempio:

- `https://wasabeef.github.io/AgentNote/dashboard/`

## Configurazione del deploy

`packages/dashboard` mantiene il default locale `/`, quindi il tuo Pages workflow dovrebbe iniettare i valori specifici del repo al momento del build:

- `SITE=https://<owner>.github.io`
- `BASE=/<repo>/dashboard`
- `PUBLIC_REPO=<owner>/<repo>`
- `PUBLIC_REPO_URL=https://github.com/<owner>/<repo>` (opzionale)

Così il package resta generico ma produce comunque gli URL pubblici corretti per ogni repository.

## Flusso consigliato

1. Attiva `dashboard: true` nella GitHub Action.
2. Ripristina `gh-pages/dashboard/notes` in `packages/dashboard/public/notes`.
3. Su `pull_request` (`opened`, `reopened`, `synchronize`), riscrivi i note files di quella PR e persisti il risultato in `gh-pages`.
4. Su `push` verso `main`, marca le PR notes corrispondenti come `merged`, ricostruisci il sito e fai deploy dell'URL condiviso.

In questo modo `main` resta pulito dal JSON generato, mentre il Dashboard condiviso può comunque mostrare la cronologia delle PR aperte e mergeate.

## Primo deploy

Un repo nuovo può accumulare Dashboard data prima del primo deploy in production, ma la URL pubblica di Pages appare solo dopo il primo deploy da `main`.

## Come funziona

- `packages/dashboard/` è l'Astro app interattiva
- `packages/dashboard/public/notes/*.json` è il working bundle usato in fase di build
- `gh-pages/dashboard/notes/*.json` è il durable store della cronologia condivisa

Il repository non include sample Dashboard JSON. Un Dashboard nuovo resta vuoto finché il tuo workflow non ripristina o genera i note files.
