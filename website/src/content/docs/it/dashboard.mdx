## Cosa ottieni

Il Dashboard offre al team un’unica URL condivisa per consultare la cronologia di Agent Note.

- PR aperte e mergeate in un solo posto
- Commit, prompt e attribution senza aprire ogni singola PR
- Un’unica URL GitHub Pages, di solito sotto `/dashboard/`

Usalo quando il PR report non basta più e vuoi una vista condivisa per il team.

## Prima di iniziare

Ti serve il Dashboard solo se vuoi una vista web condivisa.

- Se i report di sessione IA dentro le pull request ti bastano, fermati a [GitHub Action](./github-action/).
- Se vuoi una vista Pages condivisa per la cronologia aperta e mergeata, continua qui.

## Setup rapido

Per prima cosa fai in modo che la Action scriva i file del Dashboard per ogni PR:

```yaml
- uses: wasabeef/AgentNote@v0
  id: agent-note
  with:
    pr_output: description
    dashboard: true
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

Questo crea o aggiorna `packages/dashboard/public/notes/*.json` per la PR corrente. Poi il workflow Pages si occupa di ripristinare, salvare e pubblicare quei file.

## URL pubblica

Di solito il Dashboard viene pubblicato qui:

- `https://<owner>.github.io/<repo>/dashboard/`

Per esempio:

- `https://wasabeef.github.io/AgentNote/dashboard/`

<Aside type="tip">
  Se pubblichi con GitHub Pages, il workflow Pages passa i valori specifici del repository per `SITE` e `BASE` al momento del build. Nella maggior parte dei repository puoi usare il workflow di esempio senza modificare `packages/dashboard`.
</Aside>

## Primo deploy

Anche un repository nuovo può accumulare dati del Dashboard prima del primo deploy in produzione. L’URL pubblica di Pages compare però solo dopo il primo deploy su `main`.

## Come funziona

- `packages/dashboard/` è l’app Astro che mostra il Dashboard
- `packages/dashboard/public/notes/*.json` è il bundle letto durante il build
- `gh-pages/dashboard/notes/*.json` è l’archivio permanente della cronologia condivisa

Il repository non include JSON di esempio per il Dashboard. Un Dashboard nuovo resta vuoto finché il workflow non ripristina o genera i file di note. In molti team le note delle PR vengono aggiornate su `pull_request`, mentre lo stato mergeato viene pubblicato da `main`.
