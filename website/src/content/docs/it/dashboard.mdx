## Cosa ottieni

Il Dashboard offre al team un’unica URL condivisa per consultare la cronologia di Agent Note.

- PR aperte e mergeate in un solo posto
- Commit, prompt e attribution senza aprire ogni singola PR
- Vedere `📝 Context` solo visuale prima dei prompt brevi quando Agent Note recupera reference o scope
- Un’unica URL GitHub Pages, di solito sotto `/dashboard/`

Usalo quando il report della PR non basta più e vuoi una vista condivisa per il team.

## Prima di iniziare

Ti serve il Dashboard solo se vuoi una vista web condivisa.

- Se i report di sessione IA dentro le pull request ti bastano, fermati a [GitHub Action](./github-action/).
- Se vuoi una vista Pages condivisa per la cronologia aperta e mergeata, continua qui.

## Setup rapido

1. Esegui una volta la configurazione del Dashboard.

```bash
npx agent-note init --agent claude --dashboard
```

2. Fai il commit di entrambi i file di workflow.

```bash
git add .github/workflows/agentnote-pr-report.yml .github/workflows/agentnote-dashboard.yml
git commit -m "chore: enable agent-note dashboard"
git push
```

3. In GitHub Pages, scegli **GitHub Actions** come sorgente.

Il workflow Dashboard generato usa `wasabeef/AgentNote@v0` con `dashboard: true` per eseguire ripristino, sync, build, upload dell'artefatto e persistenza delle note, poi pubblica la vista condivisa in `/dashboard/`. Se lo stesso job carica già un artefatto GitHub Pages, Agent Note aggiunge il Dashboard nella directory `dashboard/` di quell'artefatto invece di sostituire il sito esistente. Se un altro job o workflow pubblica già Pages, Agent Note salta la pubblicazione standalone; metti Agent Note nello stesso job prima di `actions/upload-pages-artifact` per combinare docs site e Dashboard.

## URL pubblica

Di solito il Dashboard viene pubblicato qui:

- `https://<owner>.github.io/<repo>/dashboard/`

Per esempio:

- `https://wasabeef.github.io/AgentNote/dashboard/`

<Aside type="tip">
  Se pubblichi con GitHub Pages, il workflow Pages passa i valori specifici del repository per `SITE` e `BASE` al momento del build. Nella maggior parte dei repository puoi usare il workflow di esempio senza modificare il Dashboard source.
</Aside>

## Primo deploy

Anche un repository nuovo può accumulare dati del Dashboard prima del primo deploy in produzione. Se i deploy delle PR sono consentiti, l’URL pubblica di Pages può comparire dopo il primo run `pull_request` riuscito. Altrimenti compare dopo il primo deploy riuscito da `default branch`.

## Come funziona

- `refs/notes/agentnote` resta la Source of Truth
- `wasabeef/AgentNote@v0` in modalità Dashboard scrive snapshot delle note in `gh-pages/dashboard/notes/*.json`
- GitHub Pages pubblica il Dashboard generato in `/dashboard/`

Il repository non committa esempi JSON del Dashboard. Un nuovo Dashboard resta vuoto finché il workflow non ripristina o genera i note file. `pull_request` pubblica lo stato Open e i push verso `default branch` lo sostituiscono con lo stato Merged.

## Quando non compaiono dati

Il Dashboard ora mostra indizi concreti invece di dire solo che mancano dati.

- Nessun dato: controlla che un workflow PR sia stato eseguito dopo l’attivazione di Agent Note, che il workflow Dashboard sia riuscito e che `refs/notes/agentnote` sia stato pushato.
- `?pr=<number>` non mostra dati: il workflow di quella PR potrebbe non essere ancora partito, i Pull Request Deploys possono essere bloccati dalla environment protection di GitHub Pages, oppure il numero di PR non corrisponde ai dati pubblicati.
- `?commit=<sha>` non mostra dati: quel Commit potrebbe non avere una entry Agent Note, le notes potrebbero non essere ancora state pushate, oppure lo SHA può venire da un altro repository.

Se la pagina si apre ma resta vuota, consulta [Troubleshooting](./troubleshooting/).
