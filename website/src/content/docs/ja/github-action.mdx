## クイックセットアップ

`agent-note init` で自動生成されるワークフローを追加:

```yaml
name: Agent Note
on:
  pull_request:
    types: [opened, synchronize]
permissions:
  contents: read
  pull-requests: write
jobs:
  report:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - uses: wasabeef/AgentNote@v0
```

これだけです。すべての PR に AI セッションレポートが投稿されます。

## 入力

| 入力 | デフォルト | 説明 |
|---|---|---|
| `base` | PR のベースブランチ | 比較対象のブランチ |
| `pr_output` | `description` | PR レポートの出力先: `description` / `comment` / `none` |
| `dashboard` | `false` | 現在の PR 用の dashboard snapshot を書き出す |
| `dashboard_dir` | `packages/dashboard/public` | `notes/*.json` を書き出す親ディレクトリ |
| `output` | — | レガシー: `pr_output` の旧名 |
| `comment` | `true` | レガシー: `false` で PR レポート投稿を無効化 |

### PR Description と dashboard を同時に使う

```yaml
- uses: wasabeef/AgentNote@v0
  id: agent-note
  with:
    pr_output: description
    dashboard: true
    dashboard_dir: packages/dashboard/public
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

`dashboard_dir` には `packages/dashboard/public` を渡します。Action はその配下に `packages/dashboard/public/notes/*.json` を作成・更新します。

## 出力

| 出力 | 型 | 説明 |
|---|---|---|
| `overall_ai_ratio` | number | PR 全体の AI 比率 (0-100) |
| `overall_method` | string | `line`, `file`, `mixed`, `none` |
| `tracked_commits` | number | agent-note データ付きのコミット数 |
| `total_commits` | number | PR 内の全コミット数 |
| `total_prompts` | number | 全プロンプト数 |
| `json` | string | 構造化レポート全体 |
| `markdown` | string | レンダリング済みのマークダウンレポート |
| `dashboard_dir` | string | dashboard bundle の出力先ディレクトリ |
| `dashboard_commits` | number | dashboard bundle に書き出した tracked commit 数 |

<Aside type="tip">
  レポート内のコミットハッシュは GitHub のコミットページにリンクされます。
</Aside>

## Dashboard Package

interactive な dashboard は `packages/dashboard/` に分離されています。Action はその package 配下の data bundle を更新し、実際の GitHub Pages への publish は workflow 側で制御できます。

repo 自体には sample dashboard JSON を置きません。fresh な dashboard は、Pages sync workflow が `packages/dashboard/public/notes/*.json` を復元または生成するまで空のままです。

`packages/dashboard` 自体の local default は `/` のままにしてあるので、Pages workflow 側で `SITE`、`BASE`、`PUBLIC_REPO`、必要なら `PUBLIC_REPO_URL` を渡して deploy 先を決めます。

## GitHub Pages Sync

live dashboard は event ごとに役割を分けるのがおすすめです。

- 永続データは `gh-pages/dashboard/notes/*.json` に置きます
- 各 run でその内容を `packages/dashboard/public/notes/` に復元します
- `pull_request` (`opened`, `reopened`, `synchronize`) で、その PR の現在の note set を書き直して `gh-pages` に永続化します
- `main` への `push` で、対応する PR の note を `merged` として確定し、site を build して shared な Pages URL を deploy します
- dashboard build では `SITE=https://<owner>.github.io`、`BASE=/<repo>/dashboard`、`PUBLIC_REPO=<owner>/<repo>` を明示的に渡します
- brand-new repo でも最初の production deploy 前から dashboard data 自体は蓄積できますが、public な Pages URL が生えるのは最初の `main` deploy 後です

推奨フロー:

1. `gh-pages/dashboard/notes` を `packages/dashboard/public/notes` に復元する
2. `pull_request` (`opened`, `reopened`, `synchronize`) で、同じ PR 番号の既存 note を一度消してから、その PR の現在の commit 分だけ note JSON を書き直す
3. 更新した `dashboard/notes` を `gh-pages` に書き戻す
4. `main` push では push された commit の note JSON を書き込み、対応する PR note の state を `merged` に更新したうえで、Pages artifact を build / deploy する

これで `main` を JSON で汚さずに、1 つの shared dashboard から open / merged 両方の PR 履歴を見られます。
