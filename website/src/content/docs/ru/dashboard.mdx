## Что вы получаете

Dashboard даёт команде один общий URL для просмотра истории Agent Note.

- Открытые и уже смерженные PR в одном месте
- Коммиты, prompts и attribution без необходимости открывать каждый PR
- Один URL GitHub Pages, обычно по пути `/dashboard/`

Используйте Dashboard, когда отчёта в PR уже недостаточно и команде нужен общий обзор.

## Перед началом

Dashboard нужен только тогда, когда вам нужен общий веб-интерфейс.

- Если отчётов о сессиях ИИ внутри pull request уже достаточно, остановитесь на [GitHub Action](./github-action/).
- Если вам нужен общий Pages-экран для открытой и смерженной истории, продолжайте здесь.

## Быстрая настройка

Сначала позвольте Action записывать файлы Dashboard для каждого PR:

```yaml
- uses: wasabeef/AgentNote@v0
  id: agent-note
  with:
    pr_output: description
    dashboard: true
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

Это создаст или обновит `packages/dashboard/public/notes/*.json` для текущего PR. Затем workflow Pages восстановит, сохранит и опубликует эти файлы.

## Публичный URL

Обычно Dashboard публикуется здесь:

- `https://<owner>.github.io/<repo>/dashboard/`

Например:

- `https://wasabeef.github.io/AgentNote/dashboard/`

<Aside type="tip">
  Если вы публикуете через GitHub Pages, workflow Pages передаёт значения `SITE` и `BASE`, зависящие от репозитория, во время build. В большинстве репозиториев можно использовать пример workflow без изменений в `packages/dashboard`.
</Aside>

## Первый deploy

Даже новый репозиторий может накапливать данные Dashboard до первого production-deploy. Публичный URL Pages появится только после первого deploy в `main`.

## Как это работает

- `packages/dashboard/` — Astro-приложение для просмотра Dashboard
- `packages/dashboard/public/notes/*.json` — bundle, который читается во время build
- `gh-pages/dashboard/notes/*.json` — постоянное хранилище общей истории

В репозитории нет примерного JSON для Dashboard. Новый Dashboard остаётся пустым, пока workflow не восстановит или не создаст note files. Во многих командах PR notes обновляются на `pull_request`, а merged-состояние публикуется из `main`.
