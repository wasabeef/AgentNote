### 会话文件

| 文件 | 用途 |
|---|---|
| `prompts.jsonl` | 每行一个 prompt，并带有 turn 编号 |
| `changes.jsonl` | AI 编辑过的文件，以及 post-edit blob hashes |
| `pre_blobs.jsonl` | 每次 AI edit 之前的文件状态 |
| `heartbeat` | 会话活动时间戳 |
| `turn` | 每次 prompt 递增的单调计数器 |

### Note Schema

```json
{
  "v": 1,
  "agent": "claude",
  "session_id": "a1b2c3d4-...",
  "timestamp": "2026-04-02T10:30:00Z",
  "model": "claude-sonnet-4-20250514",
  "interactions": [

  "files": [
    { "path": "src/auth.ts", "by_ai": true },
    { "path": "CHANGELOG.md", "by_ai": false }
  ],
  "attribution": {
    "ai_ratio": 73,
    "method": "line",
    "lines": { "ai_added": 146, "total_added": 200, "deleted": 12 }
  }
}
```

## `📝 Context` 的收集方式

`📝 Context` 是只用于展示的 metadata，用来帮助理解 `continue` 或 `做下一个` 这类很短的 prompt。它不会改变 attribution。

<img src="/AgentNote/images/context-dashboard-example.png" alt="Agent Note Dashboard showing Context before a short prompt" style="width: 100%; border-radius: 20px; border: 1px solid rgba(148, 163, 184, 0.18); box-shadow: 0 24px 80px rgba(15, 23, 42, 0.28);" />

- `reference` context 来自上一条 response。只有当上一条 response 包含最终 diff 中的 changed file path、file basename、code identifier 等强 structural anchor 时才会使用。
- `scope` context 来自当前 response。只有当 response 开头用 structural anchor 说明工作范围时才会使用。
- Agent Note 不会把 `yes`、`continue`、`お願いします` 这类 approval keyword 当作选择规则。如果没有 structural anchor，就不会添加 Context。
- Context 永远不会改变 `files_touched`、prompt ownership、AI ratio、attribution method，也不会影响是否保存 git note。

## Agent Note vs Entire

Agent Note 和 Entire 解决的是相关但不同的问题。

- Agent Note 将 git commit 与产生该 commit 的 AI conversation 关联起来。Entire 是更广义的 checkpoint product，包含 rewind / resume workflow。
- Agent Note 将永久记录保存在 `refs/notes/agentnote`；临时 session 文件保留在 `.git/agentnote/` 下。Entire 以 checkpoint metadata 和 hosted web application 为中心。
- Agent Note 的 Dashboard 是基于 git notes 的 optional static GitHub Pages。即使没有 hosted service，也可以继续使用 Agent Note。
- Agent Note 不尝试恢复旧的 workspace state。它专注于 reviewability：commit 为什么变化、哪些 prompt 参与其中，以及 diff 中有多少是 AI-authored。

<Aside type="tip">
  prompt 列表来自 commit 到下一个 commit 之间的对话 window，而不是整个 session backlog。Agent Note 会保留能解释当前 commit 的 planning、clarification 和 review prompt，同时裁掉被引用的旧 prompt 历史，以及没有保留到这个 commit 中的 edit turn。
</Aside>

<Aside>
  常见的 generated artifacts 会以 best-effort 的方式根据具体 signal 排除在 `AI ratio` 分母之外，例如 framework/cache path、generated filename/suffix，以及 committed file header。仅有 `build/`、`dist/`、`gen/`、`generated/`、`target/` 这类通用目录名并不足以判定为 generated。
</Aside>

## Session 生命周期

| 事件 | 发生的事情 |
|---|---|
| **SessionStart** | 创建 session 目录并写入 heartbeat |
| **UserPromptSubmit** | 追加 prompt、增加 turn counter，并轮换 logs |
| **PreToolUse Edit/Write** | 同步捕获 pre-edit blob hash |
| **PostToolUse Edit/Write** | 捕获 post-edit blob hash 和文件路径 |
| **Stop** | 记录 stop 事件（heartbeat 保持活跃 —— Stop = response 结束，不是 session 结束） |

<Aside>
  Git hooks 会在注入 trailers 或记录 notes 前，通过 heartbeat 验证 session 是否仍然新鲜。heartbeat 会在 SessionStart 和每个 prompt 时更新。
</Aside>

<Aside type="tip">
  Agent Note 会识别当前 agent。session metadata 会保存当前 adapter，因此 `show`、`status` 和 commit recording 之后都能解析正确的 transcript format。
</Aside>

## Git Notes

Git notes 不会出现在 branch 列表、GitHub UI 或 CI 中，但依然可以 push 和 fetch。

```bash
# 与团队共享
git push origin refs/notes/agentnote

# 从 remote 获取
git fetch origin refs/notes/agentnote:refs/notes/agentnote

# 查看 note
git notes --ref=agentnote show <commit>
```

<Aside type="tip">
  `agent-note init` 会配置 auto-fetch，而 `pre-push` hook 会处理 auto-push。不需要手动命令。
</Aside>
