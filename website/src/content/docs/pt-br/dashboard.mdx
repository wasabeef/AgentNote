## O que você ganha

O Dashboard dá ao time uma URL única para consultar o histórico do Agent Note.

- PRs abertas e mergeadas em um só lugar
- Commits, prompts e attribution sem abrir cada PR
- Uma única URL do GitHub Pages, normalmente em `/dashboard/`

Use isso quando o PR report não basta mais e você quer uma visão compartilhada para o time.

## Antes de começar

Você só precisa do Dashboard se quiser uma visualização web compartilhada.

- Se os relatórios de sessão de IA dentro das pull requests já bastam, pare em [GitHub Action](./github-action/).
- Se quiser uma visão compartilhada no Pages para histórico aberto e mergeado, continue aqui.

## Configuração rápida

1. Run Dashboard setup once.

```bash
npx agent-note init --agent claude --dashboard
```

2. Commit both workflow files.

```bash
git add .github/workflows/agentnote-pr-report.yml .github/workflows/agentnote-dashboard.yml
git commit -m "chore: enable agent-note dashboard"
git push
```

3. In GitHub Pages, choose **GitHub Actions** as the source.

The generated Dashboard workflow restores data from `gh-pages`, refreshes it from `refs/notes/agentnote`, and publishes the shared `/dashboard/` view.

## URL pública

Normalmente o Dashboard é publicado aqui:

- `https://<owner>.github.io/<repo>/dashboard/`

Por exemplo:

- `https://wasabeef.github.io/AgentNote/dashboard/`

<Aside type="tip">
  Se você publica com GitHub Pages, o workflow do Pages passa os valores específicos do repositório para `SITE` e `BASE` durante o build. Na maioria dos repositórios, o workflow de exemplo já resolve sem mudar o Dashboard source.
</Aside>

## Primeiro deploy

Mesmo um repositório novo pode acumular dados do Dashboard antes do primeiro deploy em produção. A URL pública do Pages aparece só depois do primeiro deploy em `default branch`.

## Como funciona

- `refs/notes/agentnote` remains the source of truth
- The generated Dashboard workflow writes shared snapshots to `gh-pages/dashboard/notes/*.json`
- GitHub Pages serves the built Dashboard at `/dashboard/`

The repository does not commit sample Dashboard JSON. A fresh Dashboard stays empty until the workflow restores or generates note files. `pull_request` refreshes open history, and `default branch` publishes the merged state.
