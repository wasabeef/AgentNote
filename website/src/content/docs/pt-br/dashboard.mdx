## O que você ganha

O Dashboard dá ao seu time uma URL compartilhada para o histórico do Agent Note.

- Navegar pelo histórico de PRs abertas e merged em um só lugar
- Inspecionar commits, prompts e attribution sem abrir cada PR
- Compartilhar uma única URL do GitHub Pages, normalmente em `/dashboard/`

Use isso quando o PR report já não for suficiente e você quiser uma visão compartilhada para o time.

## Antes de começar

Você só precisa do Dashboard se quiser uma visualização web compartilhada.

- Se os AI session reports dentro das pull requests já bastam, pare em [GitHub Action](./github-action/).
- Se quiser uma visão compartilhada no Pages para histórico open e merged, continue aqui.

## Quick Setup

Primeiro, faça a Action escrever Dashboard bundles para cada PR:

```yaml
- uses: wasabeef/AgentNote@v0
  id: agent-note
  with:
    pr_output: description
    dashboard: true
  env:
    GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

Isso cria ou atualiza `packages/dashboard/public/notes/*.json` para a PR atual. Depois, o seu Pages workflow pode restaurar, persistir e publicar esses arquivos.

## URL pública

Normalmente o Dashboard é publicado aqui:

- `https://<owner>.github.io/<repo>/dashboard/`

Por exemplo:

- `https://wasabeef.github.io/AgentNote/dashboard/`

## Configuração de deploy

`packages/dashboard` mantém `/` como default local, então o seu Pages workflow deve injetar valores específicos do repo no build:

- `SITE=https://<owner>.github.io`
- `BASE=/<repo>/dashboard`
- `PUBLIC_REPO=<owner>/<repo>`
- `PUBLIC_REPO_URL=https://github.com/<owner>/<repo>` (opcional)

Assim o package continua genérico, mas ainda produz as URLs públicas corretas para cada repositório.

## Fluxo recomendado

1. Ative `dashboard: true` na GitHub Action.
2. Restaure `gh-pages/dashboard/notes` em `packages/dashboard/public/notes`.
3. Em `pull_request` (`opened`, `reopened`, `synchronize`), reescreva os note files dessa PR e persista de volta em `gh-pages`.
4. Em `push` para `main`, marque as PR notes correspondentes como `merged`, faça build do site novamente e deploy da URL compartilhada.

Isso mantém `main` livre de JSON gerado, enquanto um Dashboard compartilhado continua mostrando histórico de PRs open e merged.

## Primeiro deploy

Um repo novo pode acumular Dashboard data antes do primeiro deploy em production, mas a URL pública do Pages só aparece depois do primeiro deploy de `main`.

## Como funciona

- `packages/dashboard/` é o Astro app interativo
- `packages/dashboard/public/notes/*.json` é o working bundle usado no build
- `gh-pages/dashboard/notes/*.json` é o durable store do histórico compartilhado

O repositório não inclui sample Dashboard JSON. Um Dashboard novo fica vazio até que o seu workflow restaure ou gere os note files.
