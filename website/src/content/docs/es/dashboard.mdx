## Qué obtienes

El Dashboard le da a tu equipo una única URL compartida para el historial de Agent Note.

- Historial de PR abiertas y merged en un solo lugar
- Commits, prompts y attribution sin abrir cada PR por separado
- Ver context solo de visualización antes de prompts cortos cuando Agent Note recupera referencia o alcance
- Una única URL de GitHub Pages, normalmente bajo `/dashboard/`

Úsalo cuando el informe de PR ya no sea suficiente y quieras una vista compartida para el equipo.

## Antes de empezar

Solo necesitas el Dashboard si quieres una vista web compartida.

- Si los informes de sesión de IA dentro de las pull requests te alcanzan, quédate con [GitHub Action](./github-action/).
- Si quieres una vista compartida en Pages para historial abierto y merged, sigue aquí.

## Configuración rápida

1. Ejecuta la configuración del Dashboard una vez.

```bash
npx agent-note init --agent claude --dashboard
```

2. Haz commit de los dos archivos de workflow.

```bash
git add .github/workflows/agentnote-pr-report.yml .github/workflows/agentnote-dashboard.yml
git commit -m "chore: enable agent-note dashboard"
git push
```

3. En GitHub Pages, elige **GitHub Actions** como origen.

El workflow generado del Dashboard usa `wasabeef/AgentNote@v0` con `dashboard: true` para restaurar, sincronizar, compilar, subir el artefacto y persistir las notas, y después publica la vista compartida en `/dashboard/`. Si el mismo workflow ya sube un artefacto de GitHub Pages, Agent Note añade el Dashboard en el directorio `dashboard/` de ese artefacto en lugar de reemplazar el sitio existente. Si otro workflow ya publica Pages, Agent Note omite la publicación standalone; coloca el paso del Dashboard antes del upload del artefacto de ese workflow para combinarlos.

## URL pública

Normalmente el Dashboard se publica aquí:

- `https://<owner>.github.io/<repo>/dashboard/`

Por ejemplo:

- `https://wasabeef.github.io/AgentNote/dashboard/`

<Aside type="tip">
  Si publicas con GitHub Pages, tu workflow de Pages pasa los valores específicos del repositorio para `SITE` y `BASE` durante el build. En la mayoría de los repositorios puedes usar el workflow de ejemplo sin tocar el Dashboard source.
</Aside>

## Primer deploy

Incluso en un repositorio nuevo, el Dashboard puede acumular datos antes del primer deploy de producción. Si los Pull Request Deploys están permitidos, la URL pública de Pages puede aparecer tras el primer run exitoso de `pull_request`. Si no, aparece después del primer deploy exitoso desde `default branch`.

## Cómo funciona

- `refs/notes/agentnote` sigue siendo la Source of Truth
- `wasabeef/AgentNote@v0` en modo Dashboard escribe snapshots de notas en `gh-pages/dashboard/notes/*.json`
- GitHub Pages sirve el Dashboard generado en `/dashboard/`

El repositorio no hace commit de ejemplos JSON del Dashboard. Un Dashboard nuevo permanece vacío hasta que el workflow restaura o genera note files. `pull_request` publica el estado Open, y los pushes a `default branch` lo reemplazan por el estado Merged.
