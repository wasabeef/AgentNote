Esta página explica la mecánica detrás del reporte: qué se captura localmente, qué se guarda como git note y cómo esos datos llegan al PR Report y al Dashboard. No necesitas recordar todo para usar Agent Note, pero este es el modelo al que conviene volver cuando algo sorprende.

Con los git hooks generados instalados, en la mayoría de los repos basta con seguir usando `git commit` y `git push`. Cursor también recupera pares prompt / response desde transcripts locales y puede seguir registrando commits cuando los git hooks no están disponibles.

## Almacenamiento

<CardGrid>
  <Card title="Capa 1: temporal local" icon="laptop">
    Datos de sesión en `.git/agentnote/sessions/`. Archivos JSONL para prompts, cambios y blob hashes. Se rotan después de cada commit. Nunca se envían.
  </Card>
  <Card title="Capa 2: Git Notes" icon="document">
    Registro permanente en `refs/notes/agentnote`. Un JSON por commit. Se puede hacer push, fetch y compartir con el equipo.
  </Card>
</CardGrid>

### Archivos de sesión

| Archivo | Propósito |
|---|---|
| `prompts.jsonl` | Un prompt por línea con número de turn |
| `changes.jsonl` | Archivos tocados por la IA con blob hashes después de la edición |
| `pre_blobs.jsonl` | Estado del archivo antes de cada edición de la IA |
| `heartbeat` | Marca temporal de actividad de la sesión |
| `turn` | Contador monotónico que aumenta con cada prompt |

## Cómo se recopila `📝 Context`

`📝 Context` es una nota solo visual para prompts cortos como "continue" o "haz el siguiente". Ayuda a entender el prompt sin cambiar la attribution.

<img src="/AgentNote/images/context-dashboard-example.png" alt="Agent Note Dashboard showing Context before a short prompt" style="width: 100%; border-radius: 20px; border: 1px solid rgba(148, 163, 184, 0.18); box-shadow: 0 24px 80px rgba(15, 23, 42, 0.28);" />

- El `reference` context viene de la response inmediatamente anterior cuando menciona una ruta de archivo cambiada, un nombre de archivo o un identificador de código del diff final.
- El `scope` context viene de la response actual cuando sus primeras líneas explican claramente el trabajo en curso.
- Agent Note no usa palabras de aprobación como `yes`, `continue` o `お願いします` como reglas de selección. Si no hay una pista concreta de archivo o código, omite Context.
- Context nunca cambia `files_touched`, prompt ownership, AI ratio, attribution method ni si se guarda una git note.

## Agent Note vs Entire

Agent Note y Entire resuelven problemas relacionados, pero distintos.

- Agent Note vincula git commits con la AI conversation que los produjo. Entire es un checkpoint product más amplio con workflows de rewind / resume.
- Agent Note guarda el registro permanente en `refs/notes/agentnote`; los archivos temporales de sesión quedan en `.git/agentnote/`. Entire se centra en checkpoint metadata y una hosted web application.
- El Dashboard de Agent Note es static GitHub Pages opcional respaldado por git notes. Puedes usar Agent Note sin hosted service.
- Agent Note no intenta restaurar un workspace state antiguo. Se centra en reviewability: por qué cambió un commit, qué prompts participaron y qué parte del diff fue AI-authored.

<Aside type="tip">
  Las listas de prompts vienen de la ventana de conversación entre commits, no de todo el historial de la sesión. Agent Note conserva los prompts útiles de planificación, aclaración y revisión del commit actual, y recorta historiales de prompts citados y turnos de edición que no sobrevivieron en este commit.
</Aside>

<Aside>
  Los generated artifacts comunes se excluyen del denominador de `AI ratio` como best effort usando señales específicas, como rutas de framework/cache, nombres o sufijos generados y headers del archivo commiteado. Los nombres genéricos de directorio como `build/`, `dist/`, `gen/`, `generated/` y `target/` no bastan por sí solos. Si tu repository commitea bundles o generated outputs que deben seguir visibles pero no afectar el AI Ratio, inclúyelos en `.agentnoteignore`.
</Aside>

## Ciclo de vida de la sesión

| Evento | Qué ocurre |
|---|---|
| **SessionStart** | Crea el directorio de sesión y escribe el heartbeat |
| **UserPromptSubmit** | Añade el prompt, incrementa el contador de turn y rota logs |
| **PreToolUse Edit/Write** | Captura el pre-edit blob hash de forma síncrona |
| **PostToolUse Edit/Write** | Captura el post-edit blob hash y la ruta del archivo |
| **Stop** | Registra el evento stop (el heartbeat sigue activo — Stop = fin de la respuesta, no fin de la sesión) |

<Aside>
  Los git hooks verifican la frescura de la sesión mediante el heartbeat y exigen datos de sesión registrables antes de inyectar trailers o registrar notes. El heartbeat se actualiza con hook events durante turns largos; las sesiones metadata-only se omiten. Una ruta de transcript por sí sola es metadata de apoyo, no datos registrables.
</Aside>

<Aside type="tip">
  Agent Note conoce el agente activo. Los metadatos de sesión guardan el adapter activo, así `show`, `status` y el registro del commit pueden resolver después el formato correcto del transcript.
</Aside>

## Git Notes

Git notes no aparecen en listados de ramas, en la UI de GitHub ni en CI, pero sí se pueden hacer push y fetch.

```bash
# Compartir con el equipo
git push origin refs/notes/agentnote

# Traer desde remote
git fetch origin refs/notes/agentnote:refs/notes/agentnote

# Ver una note
git notes --ref=agentnote show <commit>
```

<Aside type="tip">
  `agent-note init` configura el auto-fetch y el hook `pre-push` se encarga del auto-push. No hacen falta comandos manuales.
</Aside>
