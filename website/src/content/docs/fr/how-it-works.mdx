Cette page explique le mécanisme derrière le rapport : ce qui est capturé localement, ce qui devient une git note, et comment ces données arrivent dans PR Report et Dashboard. Vous n'avez pas besoin de tout retenir pour utiliser Agent Note, mais c'est le bon modèle à vérifier quand un résultat surprend.

Une fois les git hooks générés installés, la plupart des dépôts peuvent continuer à utiliser simplement `git commit` et `git push`. Cursor récupère aussi les paires prompt / response depuis les transcripts locaux et peut continuer à suivre les commits quand les git hooks ne sont pas disponibles.

### Fichiers de session

| Fichier | Rôle |
|---|---|
| `prompts.jsonl` | Un prompt par ligne avec numéro de turn |
| `changes.jsonl` | Fichiers touchés par l’IA avec blob hashes post-édition |
| `pre_blobs.jsonl` | État du fichier avant chaque édition IA |
| `heartbeat` | Horodatage d’activité de la session |
| `turn` | Compteur monotone incrémenté à chaque prompt |

## Comment `📝 Context` est collecté

`📝 Context` est une note affichée seulement pour aider à lire les prompts courts comme "continue" ou "fais le suivant". Elle n'affecte pas l'attribution.

<img src="/AgentNote/images/context-dashboard-example.png" alt="Agent Note Dashboard showing Context before a short prompt" style="width: 100%; border-radius: 20px; border: 1px solid rgba(148, 163, 184, 0.18); box-shadow: 0 24px 80px rgba(15, 23, 42, 0.28);" />

- Le `reference` context vient de la response immédiatement précédente lorsqu'elle mentionne un chemin de fichier modifié, un nom de fichier ou un identifiant de code présent dans le diff final.
- Le `scope` context vient de la response courante lorsque ses premières lignes décrivent clairement le travail en cours.
- Agent Note n'utilise pas des mots d'approbation comme `yes`, `continue` ou `お願いします` comme règles de sélection. Sans indice concret lié aux fichiers ou au code, Context est ignoré.
- Context ne change jamais `files_touched`, prompt ownership, AI Ratio, attribution method ni le fait qu’une git note soit enregistrée.

## Agent Note vs Entire

Agent Note et Entire résolvent des problèmes proches, mais différents.

- Agent Note relie les git commits à la conversation avec l’IA qui les a produits. Entire est un checkpoint product plus large avec des workflows rewind / resume.
- Agent Note stocke l’enregistrement permanent dans `refs/notes/agentnote`; les fichiers temporaires de session restent sous `.git/agentnote/`. Entire se concentre sur la checkpoint metadata et une hosted web application.
- Le Dashboard d’Agent Note est un static GitHub Pages optionnel, alimenté par les git notes. Vous pouvez utiliser Agent Note sans hosted service.
- Agent Note ne cherche pas à restaurer un ancien workspace state. Il optimise la reviewability: pourquoi un commit a changé, quels prompts sont intervenus et quelle part du diff est AI-authored.

<Aside type="tip">
  Les listes de prompts viennent de la fenêtre de conversation entre deux commits, pas de tout l’historique de la session. Agent Note garde les prompts utiles de planification, de clarification et de revue pour le commit courant, et retire l’historique de prompts cité ainsi que les tours d’édition qui ne survivent pas dans ce commit.
</Aside>

<Aside>
  Les generated artifacts courants sont exclus du dénominateur de `AI Ratio` en best effort à partir de signaux précis, comme les chemins de framework/cache, les noms ou suffixes générés et les en-têtes des fichiers commités. Les noms de dossiers génériques comme `build/`, `dist/`, `gen/`, `generated/` et `target/` ne suffisent pas à eux seuls. Si votre repository commit des bundles ou generated outputs qui doivent rester visibles sans influencer l’AI Ratio, listez-les dans le `.agentnoteignore` à la racine du repository.
</Aside>

## Cycle de vie de la session

| Événement | Ce qui se passe |
|---|---|
| **SessionStart** | Crée le répertoire de session et écrit le heartbeat |
| **UserPromptSubmit** | Ajoute le prompt, incrémente le compteur de turn et fait tourner les logs |
| **PreToolUse Edit/Write** | Capture le pre-edit blob hash de façon synchrone |
| **PostToolUse Edit/Write** | Capture le post-edit blob hash et le chemin du fichier |
| **Stop** | Journalise l’événement stop (le heartbeat reste actif — Stop = fin de réponse, pas fin de session) |

<Aside>
  Les git hooks vérifient la fraîcheur de la session via le heartbeat et exigent des données de session enregistrables avant d’injecter les trailers ou d’enregistrer les notes. Le heartbeat est rafraîchi par les hook events pendant les longs turns ; les sessions metadata-only sont ignorées. Un chemin de transcript seul est une metadata de support, pas une donnée enregistrable.
</Aside>

<Aside type="tip">
  Agent Note garde la trace de l’agent qui a produit la session. Les métadonnées de session stockent l’adapter actif afin que `show`, `status` et l’enregistrement du commit puissent retrouver plus tard le bon format de transcript.
</Aside>

## Git Notes

Les git notes sont invisibles dans les listes de branches, l’interface GitHub et CI — mais elles peuvent être poussées et récupérées.

```bash
# Partager avec l’équipe
git push origin refs/notes/agentnote

# Récupérer depuis le remote
git fetch origin refs/notes/agentnote:refs/notes/agentnote

# Afficher une note
git notes --ref=agentnote show <commit>
```

<Aside type="tip">
  `agent-note init` configure l’auto-fetch, et le hook `pre-push` gère l’auto-push. Aucune commande manuelle n’est nécessaire.
</Aside>
