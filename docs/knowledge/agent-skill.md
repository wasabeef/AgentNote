# Agent Note Skill Design

This note defines how Agent Note should use `gh skill`.

The skill is not a replacement for Agent Note itself. It is an onboarding and
operation layer that teaches AI coding agents how to map a user's natural
language request to Agent Note's public features.

## Goal

Users should not need to remember every Agent Note command, workflow input, or
diagnostic path. They should be able to ask their AI coding agent for outcomes:

- "Set up Agent Note in this repository."
- "Enable PR Report."
- "Enable Dashboard."
- "Why did this line change?"
- "Find why Dashboard is empty."
- "Exclude generated files from AI Ratio."

The skill should translate those requests into safe Agent Note actions and
checks.

## Non-goals

- Do not teach users to run internal renderers such as `agent-note pr`
  directly. PR Report is generated by the GitHub Action.
- Do not mutate git notes by default. Git notes are commit evidence, not an
  editable task log.
- Do not replace the website docs. The skill should link users to docs when a
  decision needs human review.
- Do not hide security tradeoffs. If prompts or responses may contain secrets,
  tell the user before pushing notes to a public repository.

## Skill Shape

The public skill is one focused skill:

```text
skills/agent-note/SKILL.md
```

Keep it broad enough to cover the normal user journey, but avoid splitting it
too early. A single `agent-note` skill is easier to install and remember than a
set of narrow skills.

Future split candidates:

- `agent-note-troubleshooting`
- `agent-note-dashboard`
- `agent-note-release`

Only split when the main skill becomes too long or the install audience is
clearly different.

## Format References

The Agent Skills spec and `gh skill` behavior point to a few practical rules:

- A skill is a directory with `SKILL.md` at minimum.
- `SKILL.md` requires YAML frontmatter with `name` and `description`.
- `name` must match the directory name and use lowercase letters, numbers, and
  hyphens.
- `description` is the main trigger surface. It must explain both what the skill
  does and when the agent should use it.
- `allowed-tools` is experimental and must be a space-separated string if used.
  Avoid it in the first Agent Note skill unless a specific host requires it.
- `gh skill publish skills --dry-run` should be the release gate because it
  validates directory names, required frontmatter, and install metadata.

Good public examples use the same broad shape:

- Frontmatter first.
- Short overview.
- Clear "when to use" and "when not to use" guidance.
- Step-by-step workflow.
- Safety rules.
- Examples and edge cases.
- Optional `references/`, `assets/`, or `scripts/` only when the main
  `SKILL.md` would become too long.

For Agent Note, start with no bundled scripts. The skill should teach the agent
which Agent Note command, workflow setting, or diagnostic path to use; it should
not become a second implementation of Agent Note.

## Current Skill Layout

```text
skills/
└── agent-note/
    └── SKILL.md
```

Add references only when the main skill becomes too long or starts mixing
unrelated workflows:

```text
skills/
└── agent-note/
    ├── SKILL.md
    └── references/
        ├── user-commands.md
        ├── github-action.md
        ├── dashboard.md
        └── troubleshooting.md
```

## Trigger Test Cases

Use these prompts when testing the skill description.

Should trigger:

- "Set up Agent Note in this repository."
- "Enable Agent Note PR Report."
- "Turn on the Agent Note Dashboard."
- "Why did this line change? `src/app.ts#L42`"
- "Agent Note says no tracked commits. What should I check?"
- "Exclude `dist/**` from AI Ratio."
- "Show me recent Agent Note prompts for this branch."

Should not trigger:

- "Write a Conventional Commit message."
- "Run git blame on this file without Agent Note context."
- "Review this PR for security issues."
- "Create a generic GitHub Actions workflow."
- "Open the GitHub Dashboard."
- "Explain how Git notes work in general."

Near-miss prompts are the most important test cases. The skill should trigger
for Agent Note outcomes, not merely for generic words like `git`, `PR`,
`Dashboard`, or `AI`.

## Feature Mapping

| User intent | Agent Note feature | Skill behavior | User-facing command exposure |
| --- | --- | --- | --- |
| "Set up Agent Note" | `agent-note init` | Detect likely agent, suggest explicit `--agent` values, run init only after checking repo state. | Public |
| "Set up Agent Note for Claude / Codex / Cursor / Gemini" | `agent-note init --agent ...` | Use the named agent, explain generated hook files, and remind the user to review config changes. | Public |
| "Enable PR Report" | GitHub Action inputs | Add or update the workflow that uses `wasabeef/AgentNote@v0`. Do not ask the user to run the PR renderer locally. | Public workflow setup |
| "Enable Dashboard" | GitHub Action `dashboard: true` and Pages output | Add or update Dashboard workflow settings, explain Pages requirements, and verify `permissions` / `pages` behavior. | Public workflow setup |
| "Why did this line change?" | `agent-note why <target>` | Accept `path:line`, `path#Lline`, GitHub URLs, editor URLs, and `@path` mentions copied from AI output. Run the command and summarize evidence level. | Public |
| "Show recent Agent Note history" | `agent-note log` / `agent-note show` | Use read-only commands to inspect notes and summarize relevant prompts, responses, files, and ratio. | Public |
| "Why is PR Report missing?" | GitHub Action + `refs/notes/agentnote` | Check workflow run, notes fetch, PR permissions, and whether commits have Agent Note data. | Diagnostic |
| "Why is Dashboard empty?" | Dashboard notes + Pages | Check Dashboard workflow, Pages deploy, `dashboard/notes/*.json`, PR number, and whether data exists for that PR. | Diagnostic |
| "AI Ratio looks wrong" | attribution + `.agentnoteignore` | Inspect generated files, `ai_ratio_excluded`, attribution method, and agent support tier. Suggest `.agentnoteignore` for generated bundles. | Diagnostic + public config |
| "Exclude generated files from AI Ratio" | `.agentnoteignore` | Add precise ignore patterns, explain that files stay visible but are removed from ratio math, then verify with a new report. | Public config |
| "Push / fetch Agent Note data" | `refs/notes/agentnote` | Explain notes storage, verify whether notes were fetched or pushed, and keep normal sync on the installed git hooks. | Diagnostic |
| "Is my setup healthy?" | status / workflow / notes checks | Run a read-only doctor-style checklist: hooks, agent config, notes ref, workflow, Dashboard URL, and Pages state. | Diagnostic |

## User-facing vs Internal Commands

The skill should clearly separate public commands from internal plumbing.

Public commands:

- `agent-note init`
- `agent-note status`
- `agent-note log`
- `agent-note show`
- `agent-note why`

These are the only Agent Note CLI commands the skill should present as normal
user actions.

Internal or automation-facing commands:

- `agent-note hook`
- `agent-note record`
- `agent-note pr`
- `agent-note commit`
- `agent-note push-notes`

The skill may mention internal commands when explaining architecture, but should
not make them the primary user action.

## Safety Rules

- Prefer read-only diagnostics before editing workflow or hook files.
- Ask before committing changes. Repository instructions require explicit
  confirmation for every commit.
- Preserve existing hooks and workflows. If a hook or workflow exists, update it
  carefully rather than replacing it blindly.
- Never infer AI-authored files from shell-only evidence. Follow Agent Note's
  recorded note data.
- When troubleshooting missing data, distinguish "no tracked commits" from a
  true `0%` AI Ratio.

## Installation Direction

Once the skill exists under `skills/agent-note/SKILL.md`, validate the public
skill directory with:

```bash
gh skill publish skills --dry-run
```

Example install commands:

```bash
gh skill install wasabeef/AgentNote agent-note --agent codex --scope user
gh skill install wasabeef/AgentNote agent-note --agent claude-code --scope user
gh skill install wasabeef/AgentNote agent-note --agent cursor --scope project
gh skill install wasabeef/AgentNote agent-note --agent gemini-cli --scope project
```

Use `--scope user` when the user wants Agent Note guidance across repositories.
Use `--scope project` when the repository should carry the skill for the team.

## Relationship To `gh extension`

`gh skill` and `gh extension` solve different problems.

- `gh skill`: teaches AI agents how to use Agent Note from natural language.
- `gh extension`: gives humans a `gh agent-note ...` command surface.

Start with the skill because it is lighter and directly supports the desired
natural-language workflow. Consider a future `gh-agent-note` extension for
doctor-style commands such as `gh agent-note doctor` or
`gh agent-note dashboard open`.
